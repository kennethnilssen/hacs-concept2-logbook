## Threat model summary

- **Scope:** this integration only reads the authenticated user's own Concept2
  Logbook results and public challenge data. It never talks to any host other
  than the Concept2 API, and never issues a write-capable request regardless of
  what scope the supplied token actually carries (see the scope caveat below).
- **Credentials (updated 2026-07-11, D5):** authentication is a personal access
  token, generated by the user at their own Concept2 profile (Edit Profile →
  Applications → Concept2 Logbook API) and pasted into the config flow - no OAuth
  client registration, no client secret, no redirect URI. This replaces the
  OAuth2 Authorization Code design originally built (deferred to v1.x); see D5 in
  the design doc for the full rationale. The token is held only in Home
  Assistant's own config-entry storage - the same mechanism and protection level
  as every other HA core integration's credentials, not something separately
  encrypted by this integration. At-rest protection beyond that depends on the
  host's file permissions/disk encryption, which is outside this integration's
  control.
- **Scope - an honest limitation, not a guarantee (D5):** Concept2's personal
  access token mechanism does not expose its scope to the application using it.
  Under the OAuth2 design this integration used to have, the code explicitly
  forced `user:read,results:read` on every token request - a code-enforced
  guarantee. That control does not exist for a personal access token: whatever
  scope Concept2's own page grants when the user clicks "generate" is what the
  token carries, for as long as it's valid, and this integration has no way to
  inspect, restrict, or verify it. This integration still only ever *makes*
  read-only calls (no write-capable method exists in `api.py`), so it cannot
  itself write to Concept2 regardless of the token's actual scope - but a reader
  should understand this is weaker than the OAuth2 design it replaced, not
  equivalent to it.
- **Revocation:** users can revoke the token at any time from that same Concept2
  profile page; Home Assistant will then trigger its standard reauth flow on the
  next failed poll.
- **Logging:** the token and e-mail addresses/other PII are never logged, printed,
  or committed. Diagnostics exports redact identifiers using Home Assistant's
  built-in redaction helper.

## OWASP Top 10 (2021) alignment

| OWASP | Measure |
|-------|---------|
| A01 Broken Access Control | Read-only by design (no write-capable code path exists); no privileged operations; per-user personal access token |
| A02 Cryptographic Failures | TLS-only endpoints; tokens via HA's standard config-entry storage; no secrets in repo, logs, or diagnostics |
| A03 Injection | No dynamic query construction; all params URL-encoded via aiohttp; API responses validated before use |
| A04 Insecure Design | Least privilege; polling over inbound webhooks in v1 |
| A05 Security Misconfiguration | Pinned dependency versions in manifest; CI validation (hassfest, HACS action); GitHub secret scanning + push protection enabled on the repo |
| A06 Vulnerable & Outdated Components | Dependency versions pinned; GitHub vulnerability alerts enabled (verified via the GitHub API, not just asserted); minimum HA core version tracked deliberately |
| A07 Identification & Auth Failures | Reauth flow triggered automatically on token revocation/expiry (401 -> `ConfigEntryAuthFailed`); user-generated token, not application-managed - see the scope caveat above |
| A08 Software & Data Integrity | No runtime code download; dependency review. **Updated 2026-07-12:** `v0.2.3-alpha` is the latest release, and the first not marked a GitHub pre-release (auth, full history sync, and real sensor data confirmed working against a real account). Still explicitly `alpha`, not the "manual testing complete" v1.0.0 release that Gate 5 refers to - see README/CHANGELOG. |
| A09 Logging & Monitoring Failures | Structured logging with no tokens/PII; diagnostics redaction via HA's `async_redact_data` helper |
| A10 SSRF | API base URLs are hardcoded constants; no user-supplied URLs are ever fetched |

Full rationale for each row lives in `concept2-ha-integration-design.md` §4.5.

## Reporting a vulnerability

Please use this repository's private vulnerability reporting (Security tab →
"Report a vulnerability") rather than a public issue. Do not include real
tokens, secrets, or personal data in any report.
